[StorageKey] (8) Define opaque origin behavior

There's a need to keep StorageKey and BlinkStorageKey partial sync, but
as each organically grew this has gotten lost. As we wrap up and launch
storage partitioning, we should leave this code in a good state.

We should only maintain storage key states that provide a meaningful
distinction. Right now, there isn't a context in which we need to
distinguish between an opaque origin with kCrossSite or kSameSite
ancestor_chain_bits as there is no circumstance in which a nested opaque
context could be identical to an outer context with some other context
in between them. We should require that for any opaque origin kCrossSite
must be set.

This caps off a long set of CLs and *should* fully define StorageKey
behavior.

(1) Organize public functions and map work
(2) Rename CreateFirstParty constructors
(3) Rename CreateWithNonce constructors
(4) Rename Create constructors
(5) Organize private functions
(6) Consolidate validation
(7) Remove old test constructors
(8) Define opaque origin behavior
