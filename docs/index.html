[SuperSize Viewer] Fix DEX string literal oddities.

This CL addresses 3 DEX sting literal symbol oddities found in SuperSize
Viewer:
1. The path for these had a redundant "class" layer consists of the
   symbol's name again, e.g. 'path_to/Foo.java/"foo string"'

     path_to
       Foo.java
         "foo string" (as class)
           "foo string"

   There might have been an omission in the .size / .sizediff data;
   the literal was directly associated to the Java file while without
   a class. We can fix this later -- but for now it suffices to get:

     path_to
       Foo.java
         "foo string"

2. String literals with "." get truncated. E.g., '"Long_string[...]"'
   becomes ']"'.
3. String literals with " " get truncated. E.g., '"This has space"'
   becomes '"This'.

These oddities arise from Caspian processing DEX string literal as DEX
"other", symbols, for which "." and " " have semantic meanings.

Details:
* (Refactoring) Add {BaseSymbol,Symbol.DeltaSymbol}.GroupingPath(), and
  use it in TreeBuilder::Build() to produce input for GroupedPath.
  * Also move |kNoPath| from tree_builder.cc to model.cc.
* Symbol::DeriveNames(): Move IsStringLiteral() case before IsDex()
  case so that DEX string literals are processed using existing string
  literal logic.
* TreeBuilder::JoinDexMethodClasses(): Main loop: Identify |child| as
  string literals by checking whether |child->id_path.path| starts with
  '"'.
* Add TreeBuilderTest.TestJoinDexMethodClassesStringLiteral.
