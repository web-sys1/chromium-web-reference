Modify FrameThrottlingController API for ambient animations use case.

For ambient mode (screensaver) animations, we want to throttle the
frame rate to save power when the device is in battery mode. To do this,
we want to go through ash's FrameThrottlingController, which already
supports manual frame rate throttling (those requested explicitly by
a client). There are a few things missing though for our case use:
* We need to be able to throttle NON_APP windows (the ambient one
  falls under this ash::AppType).
* Ambient animations may require a different frame rate than the
  hard-coded default (20 fps). It depends on how much motion is in
  the animation theme that's been chosen. The caller should be able
  to choose, but also not harm the smoothness of other UIs that may
  concurrently be getting throttled.
* Add a few basic unit tests for the FrameThrottlingController.
