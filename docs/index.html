Original change's description:
> ozone/wayland: provide context for web popup menus.
>
> HTML select tag uses WebPagePopup class that further creates a
> standalone Widget without a parent. That is, the bounds that are used
> for that new Widget are in Screen and that works nicely with X11 on
> Linux, but not with Wayland.
>
> For Wayland, any auxiliary windows such as menus, tooltips, popups
> must have a parent window so that Ozone/Wayland can position them
> correctly using local surface coordinates of a parent.
>
> Prior to this CL, Ozone/Wayland used to use a currently active window,
> which does not sound as very correct solution. That is, during
> construction of the WebPagePopup, the window that had a corresponding UI
> event can loose its focus (Wayland compositor decides so) and another
> window (if exists) can gain the focus. Thus, Ozone/Wayland will choose a
> wrong parent window.
>
> To eliminate this problem, provide a context via the Widget::InitParams
> as it is done for tooltips.
>
