Wrong IDS_FOO used in BookmarkEditorView::GetMenuModel

This causes a DCHECK to be hit on Lacros, Ash/Chrome web browser and
Chrome/Linux:

> DCHECK: [simple_menu_model.cc(93)] command_id != string_id (6976 vs. 6976)

Basically, BookmarkEditorView::GetMenuModel() uses exactly the pattern
that SimpleMenuModel::AddItemWithStringId() enforces not to.
See:

> ui::SimpleMenuModel* BookmarkEditorView::GetMenuModel() {
>  if (!context_menu_model_.get()) {
>   context_menu_model_ = std::make_unique<ui::SimpleMenuModel>(this);
>   context_menu_model_->AddItemWithStringId(IDS_EDIT, IDS_EDIT);
>   context_menu_model_->AddItemWithStringId(IDS_DELETE, IDS_DELETE);
> (...)

and

> void SimpleMenuModel::AddItemWithStringId(int command_id, int string_id) {
> // Prevent this dangerous pattern:
> //   model->AddItemWithStringId(IDS_FOO, IDS_FOO);
> // This conflates string IDs with command IDs, which are separate namespaces.
> // Sometimes this is an accident where this is meant:
> //   model->AddItemWithStringId(IDC_FOO, IDS_FOO);
> (...)
> DCHECK_NE(command_id, string_id);

This CL fixes it by introducing new local IDC entries to the specific
scenarios of bookmark editor menu items, and use them accordingly.
