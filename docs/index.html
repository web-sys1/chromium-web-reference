Positioning of default button handled by platform-specific code.

The current utility that provides the value for whether a platform
prefers leading or trailing default buttons is in views, but was needed
in code that was not dependent on Views. We initially solved the problem
by creating a virtual function in FeaturePromoControllerCommon that was
implemented by BrowserFeaturePromoController (which is Views-dependent).

This CL pushes the decision of where to place default buttons down into
platform specific implementations of HelpBubbleFactory. This means that
the HelpBubbleParams::buttons field no longer guarantees ordering of
buttons; however, we do guarantee the order of non-default buttons will
not change.

We add logic to HelpBubbleView to ensure that there is only one default
button and that it is correctly placed, and then add tests to verify
that the non-default button order is not shuffled on each platform.

Finally, we simplify some of the test logic to use the new
InteractionTestUtil* classes instead of sending events directly to
Views.
