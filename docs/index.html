[sync] Avoid START_DEFERRED if not actually deferred

The specific behavioral change affects a short-lived transition
between SyncServiceImpl::TryStart() and
SyncServiceImpl::TryStartImpl(), where the first posts a tasks to
run the second.

During this short interval, the transport is not really deferred
and using INITIALIZING appears more appealing.

In practice, most calling sites do not distinguish between
START_DEFERRED and INITIALIZING.
