[VL] Move base::Feature for :visited 404s to //components/history/core

We originally wanted the base::Feature to be generated from a
RuntimeEnabledFlag. The benefit of making a runtime-enabled feature
(since we are not running an origin trial) was to have the options for
UI-based control via an about:flag and experimental-web platform
features[1].

However, //components/history/core needs to reference the flag and
cannot depend on //third_party/blink (History is a layered component and
/core is used by all platforms, including iOS, which cannot have direct
or transitive dependencies on Blink). As such, this CL moves the
base::Feature to //components/history/core.

To complete that move, this CL creates a new component() BUILD target
for history features, to avoid COMPONENT_EXPORTing the whole public API
of //components/history/core. To make the new target as small as
possible, this CL moves kTopSitesNumber to a new constants file (this
move breaks the dependency on top_sites_impl.h and all its
dependencies). That new component() BUILD target is then depended on by
BUILD targets in //content and //chrome.

Unfortunately, introducing the new target means cleaning up this flag
will require more steps if no other flag is introduced in the meantime
that needs this setup. On the other hand, this setup may be necessary
for future history base::Features, and having it around would reduce
startup costs for those features.

Since we can't use the blink::features flag, we can't control the whole
of our code with one flag, and the UI-based control methods provided by
the RuntimeEnabledFeature then become dangerous, since any base::Feature
code gates are not activated by the RuntimeEnabledFeature. Thus, this CL
also removes the RuntimeEnabledFeature.

I also took this opportunity to make the name of the feature more
specific, accurate, and concise.
