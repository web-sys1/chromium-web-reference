Split installer paths into a separate target

Both CECA and it's client code need to know of the installation paths.
The former to install the app and the latter to dial it. Separating this
code out into a target which may be depended upon by both libraries is
necessary to avoid exposing GetInstallDirectory in the client library
directly.

Exposing GetInstallDirectory in the client library is problematic
because it's implementation is dependent on the process' architecture.
If a 32-bit O4 on Windows calls this method, it would wrongly assume
that the correct CECA installation directory is descendant from "Program
Files (x86)".

A follow-up CL will add methods to the client library which will locate
and dial an existing installation of CECA.
