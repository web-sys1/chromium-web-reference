[ios] Fix compilation with C++20 standard

When passing -std=c++20, clang returns 1 when `__has_feature(modules)`
is checked, even if -fno-modules is passed. The header generated by
the swiftc compiler uses `@import` syntax when it detects that the
support of modules is available which breaks the compilation of any
file that include the generated header without correctly enabling
the support of modules.

Enabling the support of modules is currently not possible because
1.  the libc++ used by Chromium does not include a module.modulemap file
2.  ToT clang appears to have problem with Objective-C code when modules
    are enabled (the selector appears as unavailable)

This CL introduces an post-processing hack in swiftc.py that replaces
the call for `__has_feature(modules)` by 0 in the header generated by
the swift compiler (aka pretending that support for modules is not
available).
