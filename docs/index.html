Revert "ozone/wayland: Fix gbm buffer format modifiers handling"

This reverts commit c3f67f40f60907f48912531c2957ae87f61432ad.

Reason for revert: modifiers shouldn't be used on systems that
do not support any modifiers. The CL passes DRM_FORMAT_MOD_LINEAR
regardless whether DRI supports them. This results in the following
error - 

Modifiers specified, but DRI is too old
[109627:109627:0326/130318.959438:ERROR:gbm_pixmap_wayland.cc(65)] Cannot create bo with format= BGRA_8888 and usage=SCANOUT
[109627:109627:0326/130318.959640:ERROR:gpu_memory_buffer_factory_native_pixmap.cc(257)] Failed to create pixmap 3072x2048, BGRA_8888, usage SCANOUT


Original change's description:
> ozone/wayland: Fix gbm buffer format modifiers handling
>
> When Wayland Ozone backend is requested to create a native pixmaps,
> ie: GBM buffer objects, it is supposed to pass in the format modifiers
> for that buffer, which are provided by zwp_linux_dmabuf protocol
> extension. The way GbmPixmapWayland currently does that results in (at
> least) issues when dealing with video-related buffers, as described
> below:
>
> Empty format modifiers array is used in some cases, which ends ups
> calling into gbm_buffer_create(), that fails to allocate the buffer for
> some formats/usages, eg: SCANOUT_CPU_READ_WRITE, SCANOUT_VDA_WRITE. To
> fix it, this patch ensures non-empty modifiers list is passed in, even
> when buffer usage does not imply in GBM_BO_USE_LINEAR (as in these cases
> mentioned above), and DRM_FORMAT_MOD_LINEAR is used in these cases [1].
>
> [1] https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/include/uapi/drm/drm_fourcc.h#n422
