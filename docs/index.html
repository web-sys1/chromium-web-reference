Guard additional members with locking in ScreenCapture

This CL modifies mNativeDestructionLock to have an additional
responsibility of synchronizing access to members that need to be
accessed on the background thread and the desktop capturer thread. In
upcoming CLs, the members which this CL adds GuardedBy to will need to
be accessed by both threads. For example, when starting capture, the
MediaProjection API can call into MediaProjection.Callback very soon,
even before ScreenCapture#startCapture finishes.

Also, in following CLs to e.g. handle resizing/rotating of the capture,
the code becomes more complex so it's nice to have the locking
programmatically checked.
