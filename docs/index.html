[Webium] Implement client-side metric names mapping

Design Doc: [1].

This CL introduces a generic `MetricsNameMapper` to intercept and map
subprocess metrics before they are merged into the browser process.

Histograms from child processes are normally merged directly into the
browser process's global `StatisticsRecorder`. This new mapping
mechanism adds the ability to intercept and modify these histograms by
renaming or dropping them based on their exact metric name, without
hardcoding specific process types in the underlying mapping logic.

Key changes include:
* Creating the configuration proto and core separator logic in
  `components/metrics/mapping/`.
* Implementing `MetricsNameMapper` to evaluate mapping rules based on
  exact metric names.
* Introduce webium-specific flags `kWebiumMetricsMapping` and
  `kWebiumMetricsMappingConfig`.
* Updating `SubprocessMetricsProvider` to conditionally instantiate the
  mapper according to `kWebiumMetricsMapping` and use it during
  persistent histogram merging.
* Plumbing the `IsWebiumRenderer()` flag from `RenderProcessHostImpl`
  through `HistogramController` to `HistogramSynchronizer`, enabling
  name mapping for non-persistent IPC histograms.
* Modifying `base/metrics/` APIs, including `DeserializeHistogramInfo()`
  and `MergeHistogramDeltaToStatisticsRecorder()`, to mandate an
  explicit mapping callback or naming context from their callers. This
  enforces correct per-histogram renaming and ensures no metric pipeline
  unintentionally bypasses the mapping logic.
