Fix unsequenced moves in //ash/quick_pair/repository/

Fix potential issues in http_fetcher.cc with the same object being
used for reading and for std::move()'ing from within the same
expression.

Note: this specific problem will disappear once Chromium fully switches
to C++17, due to a change introduced in the evaluation order in C++17.

This issue was found by the bugprone-use-after-move clang-tidy
diagnostics:

  ash/quick_pair/repository/http_fetcher.cc:50:
  'loader' used after it was moved
  ash/quick_pair/repository/http_fetcher.cc:52:
  move occurred here
  the use and move are unsequenced, i.e. there is no guarantee about the
  order in which they are evaluated
