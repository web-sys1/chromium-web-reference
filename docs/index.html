[Editing] Use DOM positions for InsertText caret + whitespace cleanup

Under EditingUseDomPositionApi, replace VP/layout-based canonicalization
in InsertTextCommand::DoApply with DOM-only equivalents:

- Read start_position from EndingSelection().Start() instead of
  EndingVisibleSelection().Start().
- Skip MostBackwardCaretPosition() on start_position itself, so the
  user's caret is not snapped to a different inline descendant before
  insertion.
- Replace the single-sided pre-insert sweep
    DeleteInsignificantText(start, MostForward(start))
  with a two-sided sweep over [MostBackward(start), MostForward(start)],
  used only as the cleanup range. start_position is tracked via
  RelocatablePosition across the deletion so the caret is preserved
  even when surrounding whitespace nodes get removed.
- Skip the post-cleanup IsVisuallyEquivalentCandidate ->
  MostForwardCaretPosition fixup (it would re-canonicalize the position
  via layout and undo the DOM-only choice).

The widened cleanup range removes both leading and trailing unrendered
whitespace around the insertion point in one pass, matching the spec
intent without canonicalizing the caret itself. MostBackward and
MostForward are retained here only to bound the cleanup range -- they
walk through display:none, visibility:hidden, empty inline wrappers,
and fully-collapsed whitespace-only text nodes, which a pure DOM walk
would miss.
