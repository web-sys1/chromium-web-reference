blink: adds ability to cache ElementData by string

This consists of the following:

Adds support to ElementDataCache for caching ElementData by
string.

Modifies HTMLToken to use a single buffer (HTMLAttributeBuffer)
for all the attributes so that it can easily be converted to a
string.

HTMLAttributeBuffer is backed by a single UCharLiteralBuffer that
contains all the name/value pairs sequentially. In addition it
stores a vector containing the ranges (both into the document
and into the buffer).

AtomicHTMLToken changes to lookup from the cache using the
string (assuming there is a cache) otherwise create the
ElementData directly.

HTMLStackItem changes from keeping an AttributeList to storing
the ElementData.

There are a couple of cases where the ElementData is modified
after the AtomicHTMLToken is created. I don't think these
commonly occur.

No doubt this is a bit of tradeoff between size and execution
time. This should be short lived though, as the cache is dropped
shortly after parsing:

https://source.chromium.org/chromium/chromium/src/+/main:third_party/blink/renderer/core/dom/document.cc;l=6879;drc=a957a4ec4ee51f98e72501d301fab0c1cd40330b

In the short term I kept around HTMLToken::Attribute with a
way to convert from HTMLAttributeBuffer to it. This is to
lessen the number of changes (which is already far too big).
Assuming this lands, I'll do a follow on patch to remove this
and updating the places relying on it.
