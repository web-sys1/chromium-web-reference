Original change's description:
> Fix bug with tab-modal dialog positioning with the side panel.
>
> Opening the side panel shrinks the web contents area, but the side panel
> state is linked to that specific tab. Switching tabs to one where the
> side panel is closed queues a layout to expand the web contents area.
> However, if a tab-modal dialog is open in the new tab, it will be
> re-positioned synchronously in response to the tab switching event,
> which means it gets its new position before the layout happens and will
> incorrectly position itself to accommodate the side panel.
>
> To fix this, this patch recalculates the layout in BrowserViewLayout::
> WebContentsModalDialogHostViews::GetDialogPosition before calculating
> the dialog position. It does not do actually apply the layout, since
> that would be very expensive and would probably be overwritten by the
> queued layout anyway.
>
