Revert "Add a flag + SWA Config for chrome://media-app to handle audio."

This reverts commit 0c8376c06420c9f8fe5fc29ce3b14a2e65d07a31.

Reason for revert: We could not negotiate a clean 2 apps 1 icon path, and leadership did not want 2 apps 2 icons; so we're going to implement this functionality using the multi window flag (and accepting the temporary regression of newly opened files not being piped to existing windows). See https://docs.google.com/presentation/d/1m1ha9nKjRQ6mBmIe9Ss8wMCmUzqG0T8255rTFmj7Ors/edit#slide=id.geed2549801_0_5 for more information

Original change's description:
> Add a flag + SWA Config for chrome://media-app to handle audio.
>
> Flag is initially disabled, so the SWA is not presented to users.
>
> A new SystemAppType and manifest is used so that the audio "skin" of the
> app can have different icons and windowing scope. We initially use the
> same icons as the existing audio player Chrome app.
>
> To avoid colliding, a start url of `chrome://media-app/audio/` is used.
>
> This CL adds the app manifest with file handlers, and skips install of
> the old Chrome app to prevent it being added as a file handler. Since
> all the same files are supported (and it's a distinct "app" as far as
> file handlers are concerned), there doesn't seem to be a need for the
> extra complexity in file handler processing that was needed for Gallery.
>
