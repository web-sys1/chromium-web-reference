[//gpu] Reenable SharedImageSupportScanoutOnOzoneOnlyIfOverlaysSupported

We have fixed the issue with the check being enabled, which was that we
were checking a field that was only set in the browser process and so
effectively unconditionally turning off overlay support.

In addition to tightening
SharedImageCapabilities::supports_scanout_shared_images on Ozone,
turning on this feature fixes the race condition described in
crbug.com/372630273. This is because the value that is propagated from
the browser is correct at the time of its propagation, and thus if
overlays are not supported, SharedImageCapabilities will correctly
report that they are not supported independent of the initialization of
`supports_dma_buf_` in the GPU process.

Finally, turning on this feature also will turn on
ExternalVkImageBackingFactory's restricting SCANOUT support to Fuchsia
[1].

[1] https://source.chromium.org/chromium/chromium/src/+/main:gpu/command_buffer/service/shared_image/external_vk_image_backing_factory.cc;l=49-57?q=external_vk_image_back&ss=chromium
