Leak job handle when launching Isolated Browser

The Job object is used to ensure that the parent process (stub) always
outlives the isolated browser process.

However, since the Job object was stored in a ScopedHandle in the
IsolatedBrowser class, immediately this object went out of scope, the
Job would be closed, and all the processes terminated. This happened in
`ChromeMainDelegate::BasicStartupComplete`. This meant that the exit
code of the isolated browser was never returned to the shell, as the
process simply immediately terminated.

This CL fixes this by leaking the handle to the Job. This is still fine
because the Job itself will be closed when the stub process terminates,
meaning that there is still a guarantee that the stub outlives the
isolated browser - which is the reason for the Job existing in the first
place.

Because the Job handle is no longer kept, there is no need for the
IsolatedBrowser class at all, so the code can be significantly
simplified to simply return a base::Process representing the isolated
browser.
