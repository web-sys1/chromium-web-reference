Drop Wallet private passes on devices without re-auth

Wallet private passes are only supported on devices with re-auth
support. In particular, this means that the feature is not supported on:
- Linux, where DeviceAuthenticator doesn't work
- Devices that support neither biometric (hardware limitation) nor
  OS password/lockscreen re-auth (because the user didn't configure a
  password).
Note that the re-auth availability can change, e.g, if the fingerprint
sensor breaks or the OS password is removed.

This CL implements logic that removes Wallet private passes if re-auth
is not possible. This happens whenever data is loaded into the
EntityDataManager.

The re-auth availability is derived from DeviceAuthenticator::
CanAuthenticateWithBiometricOrScreenLock(). Since DeviceAuthenticator is
tied UI [1], it's not directly available in the data manager.
Instead, the info is injected into the data manager from the client.
This can lead to edge cases in settings on mobile, where - if the user
manages to navigate there without initialising a client - passes are
shown. However, what's important is that filling is prevented.

Note that:
- As of this CL, saving Wallet private passes is still supported on
  devices without re-auth. This will be addressed in a follow-up.
- For local private passes, nothing changes. They remain supported on
  devices without re-auth.

[1] https://source.chromium.org/chromium/chromium/src/+/main:chrome/browser/device_reauth/chrome_device_authenticator_factory.h;l=36;drc=fa29eb59b7b46bda1557299a1645ddf3911c69bf
