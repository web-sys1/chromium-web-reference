Avoid unconditional strong references to scroll-linked animations

We currently store strong references to animations that attach to
ScrollTimelines. The are only two ways for an animation to leave the
strong set: (1) if the timeline of the animation is changed, and
(2) via Animation::Disposed (GC finalizer). This is problematic, because
ScrollTimelines are strongly referenced by their associated scroll
source (see Node::RegisterScrollTimeline) while any animation is
attached. Hence, as long we don't switch the timeline or GC the
source Node, scroll-linked animations can *never* be garbage collected
once they are attached to a ScrollTimeline, no matter how hard they are
canceled and forgotten about by JS.

AnimationTimeline already has a system for keeping relevant animations

animations that need updates. This should be sufficient for
ScrollTimeline as well, so we can just remove ScrollTimeline::
scroll_animations_ and use AnimationTimeline::GetAnimations
whenever we need to iterate through the animations attached to the
ScrollTimeline.
