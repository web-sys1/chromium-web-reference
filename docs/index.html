Log unexpected Origin headers on renderer-initiated navigations only

https://chromium-review.googlesource.com/7617635 added
DumpWithoutCrashing calls when there are unexpected headers on
navigation requests, including Origin headers, towards the goal of
eventually killing misbehaving renderers that provide
unexpected/non-allowlisted headers. The Origin header logging in that
CL was not limited to renderer-initiated navigations, and it was
firing in at least one browser-initiated code path (going Back to a
POST navigation that redirected to a GET, which uses the `Origin:
null` header stored on the navigation entry).

This CL limits the Origin header validation code to renderer-initiated
navigations and adds a regression test checking that it doesn't dump
in this case (which will later be converted to a check that the
renderer isn't killed).

This whole adventure has surfaced that NeedsHTTPOrigin() may be
over-simplifying when it claims that GET/HEAD requests shouldn't have
Origin headers; there do seem to be GET requests on which we send an
`Origin: null` header, and I need to do some more investigation on
what the spec has to say about that. We may need to end up
allowlisting `Origin: null` in AddAdditionalRequestHeaders(), but I
haven't done that yet until we collect more DWOC data and see if it's
needed.
