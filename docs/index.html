Deduplicate browser-vs-renderer code for matching content script URLs.

If opaque origins (e.g. "data:" frames) will get isolated into a
separate process, then before this CL ContentScriptTracker might return
a wrong answer (pre-r874894, with per-frame tracking).  This is because
the browser-side code (used by ContentScriptTracker) has diverged from
the renderer-side code (used by
ScriptContext::GetEffectiveDocumentURLForInjection) to determine whether
to inject a content script (based on URL matching, "match_about_blank"
and "match_origin_as_fallback" extension manifest entries).  In
particular, a relatively recent "match_origin_as_fallback" manifest key
allows injecting content scripts into data: URLs and before this CL the
browser-side code used by ContentScriptTracker didn't account for
"match_origin_as_fallback" and data: URLs.

This CL deduplicates the browser-side and renderer-side
GetEffectiveDocumentURL logic from
//extensions/renderer/script_context.cc and
//extensions/browser/content_script_tracker.cc into a new, shared
implementation in //extensions/common/effective_document_url_getter.cc.

This refactoring is not expected to change any renderer-side behavior.
There might be minor changes in how the browser-side
ContentScriptTracker works (wrt handling of "data:" URLs and handling
empty GURLs) - these changes shouldn't in practice affect externally
visible behavior of ContentScriptTracker.
