[PolicyContainer] Fix nullptr dereference

When committing a Javascript or a XSLT document, we must reuse the
policy container of the previous document.

In Blink, the `PolicyContainer` is attached to the `ExecutionContext`.
When committing a Javascript/XSLT document, a new `ExecutionContext`
for the new document is created, and we used to move the
`PolicyContainer` from the previous `ExecutionContext` into the new
one. This left the previous `ExecutionContext` without
`PolicyContainer`, and calling, for example,
`ExecutionContext::GetReferrerPolicy()` would trigger a nullptr
dereference.

It was my belief that the old `ExecutionContext` would not be used
anymore. However, this is not true. The test included in this CL shows
how `ExecutionContext::GetReferrerPolicy()` can actually be triggered
for the old `ExecutionContext`.

With this CL, we create a new `PolicyContainer` for the old
`ExecutionContext` with a copy of the previous policies. The new
`PolicyContainer` has no corresponding `PolicyContainerHost`. In fact,
any corresponding document should be detached and in practice this
`ExecutionContext` should be useless. However, having a valid
`PolicyContainer` avoids the need of nullptr checks.
