> Original change's description:
> > HighlightOverlayPainting: clamp parts to text fragment being painted
> >
> > The new highlight painter acts at the text fragment level, but most
> > highlights are based on DOM ranges, which are defined relative to a
> > pair of text nodes (see the design doc [1] for more details).
> >
> > This results in the possibility that there may be highlighted ranges
> > that are completely outside the text fragment actually being painted
> > (e.g. because they were wrapped to some other line).
> >
> > In theory this is ok, because the preprocessing algorithms can treat
> > the originating content as Just Another Layer, and when we actually
> > try to paint, we can clamp the resultant “parts” to the fragment,
> > collapsing them to empty ranges at the start or end of the fragment.
> >
> > In practice, attempting to paint text and decorations for such empty
> > ranges is a very expensive no-op, causing massive regressions in paint
> > times in scenarios like typing in a contenteditable with many spelling
> > errors (e.g. because the dictionary was set to the wrong language).
> >
> > This patch changes the ComputeParts algorithm to clamp the resultant
> > parts to the fragment, discarding any parts that have no overlap with
> > the fragment, which is equivalent to clamping and skipping them in the
> > painter but more efficient.
> >
> > We also replace the paint/spelling-errors perf test with a harder test
> > scenario that would actually have caught this bug (based on [2]).
> >
> > [1] https://docs.google.com/document/d/1S621Fkdu-KGaK6df2Exa8_5evPKj7jm889dOxvoktWo
> > [2] https://bucket.daz.cat/work/igalia/0/46.html?p=spelling&iwc=500
