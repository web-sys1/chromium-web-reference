Fix speed regression in performance.getEntriesByName

UserTiming already stores a map from name to a list of performance
entries for both marks and measures. Instead of using that map,
performance.getEntriesByName is currently fetching the whole lists of
marks and measures and then filtering the result. This causes major
slowdowns on sites that produce many marks and/or measures and which
often call performance.getEntriesByName.

In this change, I propose that Performance::GetEntriesForCurrentFrame
and Performance::getEntriesByTypeInternal should produce lists which
have already been filtered by name, if such filtering is required. In
this way, they can benefit from the versions of UserTiming::GetMarks and
UserTiming::GetMeasures which take a name parameter.
