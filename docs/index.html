this patch in range. It was not the culprit, as the revert landed in
https://ci.chromium.org/ui/p/chromium/builders/ci/Win7%20Tests%20(1)/113864/blamelist
without fixing the bot. The real culprit was likely identified in
https://bugs.chromium.org/p/chromium/issues/detail?id=1174890, since the
build was fixed at
https://ci.chromium.org/ui/p/chromium/builders/ci/Win7%20Tests%20(1)/113864/blamelist.

> > > From the original change:
> > > Periodic purge should ideally not be scheduled at all when a process is
> > > fully idle from the allocator's standpoint, that is doesn't
> > > allocate. The current current scheme achieves that, with non-trivial
> > > complexity (including managing reentrancy in the allocator).
> > >
> > > Make the code simpler, by modulating purge frequency with exponential
> > > backoff, rather than the complex mechanism we have.
> > >
> > > Another motivation is crbug.com/1167373#14, where crashes for periodic
> > > purge were found to stem from allocator-triggered reentrancy in the task
> > > scheduler. This changes no longer posts tasks from malloc() / free(),
> > > which is the root cause of the crashes.
