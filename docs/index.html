Consider path if available when matching system tickets as a user.

Keystone Registration Framework compares the existence checker path of
an existing ticket to the one provided when the KSRegistration object
was initialized to verify whether a system ticket is actually the
correct match for a lookup based on app ID. To correctly look up an app
registration, _any_ ksadmin client needs to do this. Instead of porting
KSRegistration's logic to CRURegistration, this CL ports it into
ksadmin itself, recycling the existing `-x` / `--xcpath` flag because
this refers to the same path, albeit in a new context.

An intermediate revision of this CL (not uploaded) devised a fuzzier
match that would attempt to match an existence checker path to the path
of the binary running CRURegistration, making several assumptions about
bundle structure and the ways packages apparently currently specify or
historically specified XC paths, but that became a morass and I realized
the complexity wasn't justified; CRURegistration will simply require an
existence checker path as an initialization parameter, which will not
require any new plumbing when migrating existing code because
KSRegistration also required it at initialization time.
