Always use browser-supplied IO TaskRunner for in-browser RenderProcesses

Before this CL, RenderThreadImpl::GetIOTaskRunner() exists as an
override of ChildThreadImpl::GetIOTaskRunner(). ChildThreadImpl's
version would return the browser-process-supplied TaskRunner if the
ChildProcess is running in-browser, and return
ChildProcess::current()->io_task_runner() otherwise. But
RenderThreadImpl::GetIOTaskRunner() would *always* return
ChildProcess::current()->io_task_runner().

I believe this is causing a subtle multi-year-long race condition in
RenderThreadImplBrowserTest, as described by the design doc below [1].
We can land this and verify that the test stops being flaky on the CQ,
and all should be well. However, this change has non-test consequences
as well.

https://docs.google.com/document/d/1lzyEc8g5BcfV67SDgBylJhl1_5HAVGjcXRuiD_ydZEw/edit
