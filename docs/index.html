[View] AnimationAbortHandle handles layer destruction

Currently, AnimationAbortHandle does not handle the case that the
tracked layers could be destroyed. Note that the layers tracked by
AnimationAbortHandle could be destroyed in two manners:

(1) The layer is destroyed before ~AnimationAbortHandle() is called
(2) The layer is destroyed during ~AnimationAbortHandle() when
aborting layer animations

As a result, the user of AnimationAbortHandle could cause crashes, just
as the attached issue demonstrates.

This CL fixes this issue by:

(1) Making AnimationAbortHandle an observer of the tracked layers.
(2) When the observed layer is destroyed, add the layer to an exclusive
set called `destroyed_layers_`.
(3) In ~AnimationAbortHandle(), do not access the layer points included
by `destroyed_layers_`.

Note that the animation end/abort callback may destroy multiple layers.
It is why an additional set is used to track the destroyed layers
instead of removing the deleted layers from the existing set. Otherwise,
iterating the existing set during ~AnimationAbortHandle() could be risky
