Original change's description:
> Implement `canTransferInputToViz` method in InputTransferHandler.
>
> We are by default transferring input except in following cases:
> - `mVizToken` is null: To handle cases where browser hasn't received the
>   viz side token yet, and a touch sequence might have started. This will
>   also cover the case of split window where Viz would never set viz
>   token for second window since we are only using a single
>   AInputReceiver for now due to an Android platform bug which prevents
>   us from calling AInputReceiver_release.
> - When text selection is active or IME is active, this is done to
>   prevent ordering issues between input and text selection/IME related
>   mojo methods.
> - When page is at top, this is added to allow overscroll/refresh effect
>   to be handled on Browser.
>
> P.S. We are also changing point where `input_transfer_handler_`
> intercepts events in RenderWidgetHostViewAndroid, this is to resolve
> crash happening due transferred events sitting in
> TouchDispositionGestureFilter and acks being expected to come for them.
> Call to `gesture_provider_.OnTouchEvent` was pushing events to
> TouchDispositionGestureFilter.
>
> i.e. input coming from system server directly instead of being injected
> to views.
>
> Low-Coverage-Reason: HARD_TO_TEST Requires testing touch transfer APIs
