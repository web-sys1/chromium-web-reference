android: Coalesce independent begin frame

Follow up to fling optimization. There is no throttling mechanism, so
begin frame messages can be queued up if UI thread is slow, causing
extra unnecessary work. So try to coalesce begin frame messages in this
case by posting a task to gather already pending messages.

First only do this if there is potentially more messages pending. Then
ensure the posted task and begin frame messages use the same task queue.
Finally post task with a small non-zero delay which apparently is needed
to ensure it comes after already pending mojo messages.

Use the kUserInput queue to match priority of regular begin frames.
