Original change's description:
> Reland: Enable generation of .dwarf_aranges section
>
> Note: this does NOT enable for tsan/asan as it triggers different
> behavior in llvm-symbolizer that breaks the expectations regexps. It
> also does not enable in nacl as it causes a internal compiler error.
>
> The .dwarf_aranges provides an index from address in a module to
> location of their respective DW_TAG_compile_unit or DW_TAG_partial_unit
> DIE inside the .debug_info section. This allows tools to more quickly
> look up information for a given program counter. It is enabled by
> default on GCC, but disabled on clang. According to
>
> https://gcc.gnu.org/wiki/DebugFission
>
> it adds 0.1% overhead to the file size. Sanity checking against
> the object files produced in a clean base_unittest build then doing
>
> $ find out/Debug -name '*.o' > files
> $ du -c `cat files` | tail -1
>
> We see that for symbol_level=2:
>
>   4315544 w/o .debug_aranges
>   4321372 w/  .debug_aranges (~0.14% bigger)
>
> And for symbol_level=1:
>   3794856 w/o .debug_arangs
>   3795476 w/  .debug_aranges (~0.16% bigger)
>
> Numbers are in kilobytes.
>
> For the runtime speed up, when testing retrieval of line number
> information, resolving a pc using .dwarf_aranges saved about 70ms out
> of 140ms as compared to doing a linear search of the whole .debug_info
> section.
>
> Given that linux builds are generally fast anyways, and that gcc has it
> enabled by default, this seems like a good trade.
>
