Remove separate suffix parsing and support new <ac> - <phone> rule.

The PhoneFieldParser::Parse searches for phone number fields based on
grammars. These grammars may contain FIELD_SUFFIX entries, which
corresponds to the zzz in a US phone number of format "(xxx) yyyy zzzz".
After processing all grammars, the function took another attempt at
discovering a phone suffix. This extra attempt succeeded in 0.03% of
phone number fields (i.e. 3 in 10.000). I tried incorpoating this as new
grammar rules but I am concerned about negative side effects. The only
rules that carried some traffic were:
- Area Code: <ac> Phone: <phone>
- Phone: <phone>

I have replaced suffix handling for the first rule with new grammars:
```
      // Area Code: <ac> Phone: <phone> - <suffix>
      {{REGEX_AREA, FIELD_AREA_CODE},
       {REGEX_PHONE, FIELD_PHONE},
       {REGEX_SUFFIX_SEPARATOR, FIELD_SUFFIX}},
      // Area Code: <ac> Phone: <phone> Suffix <suffix>
      {{REGEX_AREA, FIELD_AREA_CODE},
       {REGEX_PHONE, FIELD_PHONE},
       {REGEX_SUFFIX, FIELD_SUFFIX}},
```

I have not introduced a rule
```
      // Phone: <phone> - <suffix>
      {{REGEX_PHONE, FIELD_PHONE}, {REGEX_SUFFIX_SEPARATOR, FIELD_SUFFIX}},
```
because I believe that such a construct should be interpreted as
```
      // Phone: <ac> - <phone>
      {{REGEX_PHONE, FIELD_AREA_CODE}, {REGEX_PREFIX_SEPARATOR, FIELD_PHONE}},
```

I am not rolling this out behind a finch gate because these rules trigger very
rarely. They don't trigger on my end-to-end tests.
