Remove clip path cache in FragmentData

The cache was to avoid a new copy of Path to be created when
- the element having clip-path that could be path-based needed paint
  property update; and
- the clip-path didn't change.

By removing the cache, we'll create a new copy of Path in the above
case, but the performance impact will be trivial because:
1. The case is not common;
2. The clip path is either a simple shape or a reference to a
   not-too-complex clip-path (otherwise we will use mask-based
   clip-path), so creating a new Path is not expensive. Also for the
   reference case, we cache the Path in LayoutSVGResourceClipper, so
   the extra cost is just to create a path with an offset when the
   paint offset is not zero (and the cost could be avoided if we
   didn't bake paint offset into the path but this seems not worth
   it).

Removing the cache simplifies the invalidation logic.

blink::Path is based on SkPath which is internally ref-counted and
copied on write, so copying a blink::Path is trivial and
RefCountedPath is unnecessary.
