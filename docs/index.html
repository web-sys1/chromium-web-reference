Store scrolling contents cull rect on ScrollPaintPropertyNode

Previously we stored scrolling contents cull rects in PaintChunk::
hit_test_data for scrolls, which was to avoid the change of
ScrollPaintPropertyNode on scrolls, but caused trouble when we need to
find the scrolling contents cull rect associated with a scroll node in
the main thread. We could maintain a map from scroll nodes to cull
rects in PaintArtifactCompositor during layerization, but that would
add complexity and have issues when the cull rect is needed before the
scroll hit test data paint chunk.

Now (behind killswitch ScrollingContentsCullRectOnScrollNode) store
scrolling contents cull rects on ScrollPaintPropertyNode during cull
rect update. A fast path direct update is added to keep the repaint-
only PAC update optimization.

This prepares for merging sticky layers moving together within the
range of scrolling contents cull rect.

This should not cause any behavior change.

Binary-Size: The binary size bot reported "kScrollingContentsCullRectOnScrollNode" as a new mutable constant added, but that's in the generated code of runtime_enabled_features.json5 which I have no control. It's just a runtime feature as a temporary killswitch.
