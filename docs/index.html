Ensure that NativeWidgetMac respects Widget z order level

Currently, on Mac, menus that are created using the |MenuHost| are displayed under floating windows (while also still receiving input events). This is due to the underlying widget having an incorrect z-order.

The issue here is that when Widgets are created we try to determine the |EffectiveZOrderLevel| [1]. However, on Mac (under certain circumstances) we override the Widget z-order [2], which does not take into account the |EffectiveZOrderLevel|.

This change updates NativeWidgetMac to use |EffectiveZOrderLevel| so that the z order level is set appropriately according to the Widget InitParams Type.

Also added a MenuController test to prevent future regressions.

[1]: https://source.chromium.org/chromium/chromium/src/+/main:ui/views/widget/widget.cc;l=170;drc=430c7ea2e0a2e29142869f6efb112a7979ab605a
