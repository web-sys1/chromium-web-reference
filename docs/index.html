> Original change's description:
> > Fix two broken tests in tray_accessibility_browsertest.cc
> >
> > This CL addresses failures in two browser tests within
> > tray_accessibility_browsertest.cc: ShowMenuWithShowMenuOption and
> > ShowMenu.
> >
> > Two separate issues were fixed:
> >
> > 1. When a user disables Switch Access, a confirmation dialog is shown.
> > To ensure users without a physical keyboard can interact with this
> > dialog, the virtual keyboard is intentionally kept enabled. The logic
> > for this is as follows: (1) When Switch Access is enabled,
> > AccessibilityManager::OnSwitchAccessChanged enables the virtual keyboard
> > by calling ChromeKeyboardControllerClient::SetEnableFlag. (2) When the
> > user disables Switch Access,
> > AccessibilityController::UpdateFeatureFromPref shows the confirmation
> > dialog but does not yet deactivate the feature. (3) After the dialog is
> > dismissed, AccessibilityController::SwitchAccessDisableDialogClosed
> > calls DeactivateSwitchAccess(). This eventually calls
> > AccessibilityManager::PostUnloadSwitchAccess (in
> > chrome/browser/ash/accessibility/accessibility_manager.cc), which
> > restores the virtual keyboard state by calling
> > ChromeKeyboardControllerClient::ClearEnableFlag.
> >
> > In the browser tests, this dialog was not handled, causing the virtual
> > keyboard to remain enabled after Switch Access was toggled off. Since
> > the virtual keyboard is an accessibility feature, this caused
> > IsMenuButtonVisible() to incorrectly return true, failing assertions in
> > both ShowMenu and ShowMenuWithShowMenuOption tests.
> >
> > The fix is to disable the confirmation dialog for the test suite. This
> > is a standard pattern for tests that involve such dialogs, allowing the
> > test to verify the final state after the feature is disabled without
> > having to interact with intermediate UI. For example,
> > AccessibilityFeaturePodControllerTest.FeatureTileAllFeaturesToggled in
> > ash/system/accessibility/accessibility_feature_pod_controller_unittest.cc
> > disables dialogs for both Autoclick and Switch Access. Manually
> > disabling the virtual keyboard in the test is not ideal as it would be
> > working around the product logic rather than testing it in a controlled
> > environment.
> >
> > 2. The EnableDictation() helper function was using a method that only
> > toggles an already-enabled Dictation feature, rather than enabling the
> > feature's preference. In the ShowMenu test, this caused the
> > kAccessibilityDictationEnabled preference to remain false after
> > EnableDictation(true) was called. The accessibility menu button is only
> > visible when at least one accessibility feature is enabled, so the
> > subsequent EXPECT_TRUE(IsMenuButtonVisible()) assertion failed. This was
> > fixed by changing the helper to use
> > AccessibilityManager::SetDictationEnabled(enabled), which correctly
> > updates the underlying preference.
> >
