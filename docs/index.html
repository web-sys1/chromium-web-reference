Fuzzing coverage timing & retry tweaks.

The fuzzing coverage dashboard currently takes 4.8 hours to run all the
fuzzers, but fails to gather coverage for the MojoLPM fuzzers. This CL
is an attempt to adjust some of the timeouts and retry limits to
(hopefully) achieve better coverage collection within the same sort of
time limit.

The overall algorithm (unchanged) is:
* Try to run the fuzzer against its whole corpus, retrying
* If that doesn't work, run it against individual test cases, until
  we've gathered 'n' traces successfully, or attempted 'm' traces

This CL:
* Increases 'n' from 10 to 100. This is what seems to be needed to
  gather coverage from most of our MojoLPM fuzzers. (100 is just a
  guess). This will obviously increase the time taken for the job.
* Decreases 'm' from 1000 to 500, which will save some time but will
  likely only impact two fuzzers, so won't save much.
* Decreases the retries on the initial whole-corpus run from 2 to 1
  because there's no evidence that retries actually help anyway.
* Decreases the timeout for each individual-test-case run from 30
  minutes to 1 minute. I don't think we're currently hitting this
  timeout so this probably won't make any difference.
* Decreases the timeout for the whole-corpus run from 30 minutes
  to 20 minutes. This is being hit 67 times at the moment - it's
  not clear whether this is because these fuzzers are stalled or
  because it's just taking a long time to run them. So this is a
  bit of a gamble - this could reduce overall execution time (if the
  former) or increase it (if the latter, and we then have to run
  more test cases individually.)
