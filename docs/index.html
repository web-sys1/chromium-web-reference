Revert "Replace GetAvailableLocales() with separate ICU and pak locale lists"

This reverts commit 7254c57be68147b2c7d4b82d5d10376c133cdcb7.

Reason for revert: Appears to break tests on Mac:
https://ci.chromium.org/ui/p/chromium/builders/ci/Mac10.14%20Tests/13995/overview

Original change's description:
> Replace GetAvailableLocales() with separate ICU and pak locale lists
>
> The current GetAvailableLocales() logic uses an ICU-based list of
> locales and checks whether locales are available through ICU-based
> logic. However, most callers expect GetAvailableLocales() to return a
> list of locales that Chrome has translation strings for.
>
> This discrepancy has resulted in a few bugs:
> - The list of locales in Chrome OS's OOBE screen is huge (~350 locales)
>   compared to the list of locales selectable in the language settings
>   page (~80 locales). Most of those locales aren't selectable, or are
>   regional variations of locales we have strings for.
> - Esperanto incorrectly appears as a selectable Chrome locale in the
>   CrOS and Windows language settings pages. Selecting it is equivalent
>   to selecting English (US).
> - New locales being added to Chrome are not selectable by users.
>
> This CL replaces GetAvailableLocales() with two functions: the existing
> function is kept and is renamed to GetAvailableICULocales(), and most
> callers are replaced with a new function GetLocalesWithStrings().
>
> GetLocalesWithStrings() use kAcceptLanguageList as a base for the list
> of locales - similar to the language settings page - and uses
> CheckAndResolveLocale() to check whether we have a valid fallback for a
> given locale, using IsLocaleAvailable() (now renamed to
> HasStringsForLocale()) to check for pak files. As a result, this
> resolves all the bugs listed above.
>
> However, as there are many callers of GetLocalesWithStrings() from
> threads where I/O is prohibited, we need to modify HasStringsForLocale()
> to not use I/O - as it checks the pak files on disk. Because of this,
> this CL also introduces a way of calling CheckAndResolveLocale() and
> HasStringsForLocale() without performing I/O by generating the list of
> locales supported by the platform at build time. Using this may result
> in false positives on Android and iOS.
>
> Fixed: 1140826, 1128857
