Original change's description:
> [PWA][1/N] Defer Launch Params enqueueing until navigation commits
>
> This CL refactors `WebAppLaunchNavigationHandleUserData` to directly
> hold a `webapps::LaunchParams` object instead of just an app ID, passed
> to it via the `NavigateParams`.
>
> When an app launch triggers a navigation, the launch parameters are
> attached as user data on the `NavigationHandle`. Upon completion, the
> `WebAppUiManager` verifies the navigation status and dispatches the
> launch parameters to the tab helper's launch queue if the navigation
> successfully committed and is in-scope. Non-navigating launches (such as
> focus-existing actions) enqueue launch parameters directly. The
> `WebAppUiManager` hop is needed to access `c/b/ui/` logic. This allows
> launch params to be enqueued as soon as the navigation commits. The
> `LaunchQueue` implementation has also been simplified to support this.
>
> There is still some code like extensions which enqueue launch params
> before navigation commits, which will be modified in follow-up patches.
> The changes in this CL are strictly needed to ensure all tests pass.
>
