[Connection-Allowlist] Dedicated/Shared worker fetch intercepted by SW

When fetches are intercepted by service workers, the Connection
Allowlist check for the initiator context needs to be done in the
renderer since the SWs factory gets directly called by the renderer
and the checks in browser or initiator's URLLoaderFactory are not
invoked, as documented in
https://source.chromium.org/chromium/chromium/src/+/main:docs/connection_allowlist_design.md;l=1?q=connection_allowlist_design&sq=&ss=chromium#:~:text=and%20handling%20redirects).-,Blink%20checks,-Blink%20is%20the.

The renderer check already exists in
BaseFetchContext::CanRequestInternal(). It works for document and shared
workers but not for dedicated workers because their PolicyContainerHost
is not propagated, which has an existing TODO tracked in issue 40168691.
This CL propagates the dedicated workers' PolicyContainerHost to the
renderer so the CA check can also work for dedicated workers fetches
intercepted by SWs.
It also adds tests for both dedicated and shared worker fetch
intercept cases.

Bug: 515090596, 40168691
