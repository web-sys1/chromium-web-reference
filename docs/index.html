Fix kEncryptSyncCompat to handle only non-sync compat providers

If the only providers being registered did not support OSCrypt sync but
were flagged as in use for encryption, then Encryptors returned
via a call to GetInstance with the kEncryptSyncCompat would still
return an Encryptor that encrypts to one of these providers,
because the current code did not take into account this scenario.

This CL fixes this by making it so if no providers say they are
OSCrypt sync compatible, and a call to GetInstance is made with
kEncryptSyncCompat option specified, then the default encryption
provider is set to empty string, causing fallback to OSCrypt
sync to always occur.

This allows a future migration from OSCrypt Sync to OSCrypt Async
without needing to implement a sync-compatible provider (e.g.
the DPAPI provider, on Windows), while still allowing for a
safe rollout strategy with supported rollback during the
transition from sync to async calls.

Tests are added to verify this behavior.
