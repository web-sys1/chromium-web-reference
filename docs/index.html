Revert "Let InvalidationSet::Backing store AtomicString* instead of StringImpl*"

This reverts commit d26cb6fe579dfad082b5ad97b5b2b34f8188d66c.

Bug: crbug/1302913

Reason for revert: speculative revert to fix crbug.com/1302913.
This CL seems not the root cause, but makes chrome to modify StringImpl
more often. So more StringImpl::Release() crashes occur.

Original change's description:
> Let InvalidationSet::Backing store AtomicString* instead of StringImpl*
>
> Making string_impl refcount atomic showed a perf regression on the
> blink.css/AttributeDescendantSelector microbenchmark caused by the
> Construction of AtomicStrings in InvalidationSet::FindAnyAttribute.
>
> This CL refactors the code of InvalidationSet::Backing to store
> an AtomicString* instead of StringImpl*, which is possible because
> the InvalidationSet::Backing::string_impl_ object was always
> constructed from an AtomicString. This way
> InvalidationSet::FindAnyAttribute can pass a const reference to an
> AtomicString without constructing a new object.
>
> Latest pinpoint show an average improvement of 25.2% in the
> AttributeDescendantSelector microbenchmark.
>
