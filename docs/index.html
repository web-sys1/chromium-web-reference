Original change's description:
> [Android][Settings] Implement basic UI flow
>
> Implements basic settings search UI flow. The UI can be in 3 states:
>
> - Settings (default) state:  User browses settings
> - Search state: Starts when a user taps the fake search box and
>   enables the actual text edit. In this state, user may enter queries
>   and get the search results displayed in the fragment. Back press
>   leads back to the default state.
> - Result state: Starts when a user taps one of the search
>   results. Fragment is updated with the settings fragment that
>   contains the chosen settings. Users can alter the settings or browse
>   other settings, enter child settings under the hierarchy. User can
>   get back to the first settings fragment by back press. Back press at
>   the first fragment exits the state and returns to search state where
>   the search results were shown. Or initiating another search by
>   entering a new query also changes the current state to 'search'.
>
> SettingsSearchCoordinator defines OnBackPressedCallback to manage
> these states, and perform UI tasks upon state update. The back press
> callback gets activated when entering search state, and deactivated
> when going back to the default state. There is a bug when run in
> MultiColumn mode. This will be taken care of in a follow-up CL.
>
> Notes:
> - Actual search is not possible due to lack of the index data
> yet to be built from all the Settings Fragments. The search operation
> returns a canned results after built as
> SearchResultsPreferenceFragment.
>
> - This CL implements the first 2 states only i.e. tapping search
>   results does nothing, mainly to keep the size of changes small.
>
> - In order to make it easy to manage Fragment transition, only a
>   single new Fragment (that clears up the screen) is added to
>   backstack when entering search state. All the data are displayed in
>   a new Fragment simply replace the current one without being added to
>   backstack.
>
