views: fix off-by-one in menu scroll offset computation

MenuScrollView::OnScroll has an off-by-one error when clamping its new
visible rectangle, which caused it to be impossible for it to scroll
all the way to its own bottom - it would instead end up 1px above the
bottom. This confused the logic in
MenuScrollView::ScrollRectToVisible and caused the end-of-menu
indicators (the arrows) to flicker in and out of view.

The reason this only happens with the touchpad is because of a separate
problem (not addressed in this CL): touchpad scrolling in SubmenuView
is routed through OnMouseWheel, which assumes that the mouse wheel
events are quantized by a constant called
ui::MouseWheelEvent::kWheelDelta. For touchpads, this is not the case
(at least on Mac), and the code in OnMouseWheel generates many spurious
calls to ScrollView::OnScroll() as a result. Fixing this is left as an
exercise for some future engineer.
