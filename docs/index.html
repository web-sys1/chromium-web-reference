Revert "Remove overdraw detection logic from 2d canvas"

This reverts commit b3b2a41bb2fa2a867088b1d76cabe1f00a577efc.

Reason for revert: Memory regression: crbug.com/1244272

Bug: 1244272

Original change's description:
> Remove overdraw detection logic from 2d canvas
>
> Historical context: Overdraw optimizations were originally
> implemented to improve Chrome's score on legacy benchmarks that were
> used by the press for comparing browser performance.  Overdraw
> optimization refers to the logic that discards all pending draw
> operations whenever a single draw operation occludes the entire canvas.
> Typically, this optimization only affects animations that are not
> synchronized with the display, which was the norm before the
> introduction of the requestAnimationFrame API.
>
> The logic for determining whether a draw operation occludes the canvas
> is cumbersome. Even in the simplest early-exit scenarios it includes
> non-inlinable function calls and conditional branches on every single
> draw call. This resulted in measurable overhead that brought no benefit
> to well designed JS animation code.
>
> This CL proposes a compromise: it still performs overdraw optimization
> when the canvas is explicitly cleared, but it avoids overdraw detection
> logic on every single draw call. This way, most animations that are not
> synchronized with the display (do not use rAF) can still benefit from
> the optimization because they reset the canvas at the start of each
> animation frame. All the while, the runtime of canvas draw methods is
> slightly reduced.  As a result of this change the performance a few old
> or poorly written web pages may suffer, but code that respects best
> practices will benefit.
>
> With this change, the Canvas Lines test of the MotionMark benchmark
> (which is CPU-bound on the renderer main thread) runs about 3% faster.
>
