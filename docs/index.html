Fix BackgroundSyncManager Service Worker existence check

This CL fixes a bug in which
BackgroundSyncManager::UnregisterForOriginImpl() deletes its local
service worker registration before it's done using it.

This is done by only erasing the registration from the active set until
after StoreRegistrations() is called.

The bug is as follows:
BackgroundSyncManager has a set of active registrations it uses to
determine which origins require background syncing.

When an origin is unregistered via UnregisterForOriginImpl() all
associated registrations are removed from the set. The method then
checks if a service worker's registration still exists in the service
worker database via a call to StoreRegistrations() for each registration
id. StoreRegistrations() uses the id to look up the registration's info
in the active registration set. This look up is through the [] operator,
which causes the std::map to create an empty entry (as the entry has
already been removed).

This leads to StoreRegistrations() trying to find a registration for an
empty origin which will cause UnregisterForOriginDidStore() to receive
a kErrorNotFound, potentially skipping DisableAndClearManager() if the
registration actually does exist in the SW database.

This was found during refactor work for the associated bug (but is
otherwise unrelated).
