Reland "[ProfileMenu] Fix double trigger of the menu potential crash"

This is a reland of commit e46e5a33a625fbcaf76eb8d6ff5397ae415032c9

The initial CL was failing on bots that were running with the
`browser_tests_no_field_trial` mode - which turned off
`syncer::kReplaceSyncPromosWithSigninPromos` and therefore ignored
the BatchUpload changes which the new test rely on.
Fix: force enable the feature as the test is only useful when the
feature is enabled. Enabled at the suite level which affected 2
other tests - this is ok since the feature will be launched soon.

Original change's description:
> [ProfileMenu] Fix double trigger of the menu potential crash
>
> When opening the menu, an asynchronous request is made to get local
> data items, which allows for potentially multiple triggers before the
> first results are received, which would trigger the menu showing.
>
> In practice this should happen rarely, as the asynchronous call is
> not common.
> The fix here ignores subsequent results if the menu is already showing.
> An alternative solution could be to track whether a request is already
> on-going, but given the rarity of this event to happen, the current
> solution seems appropriate and clearer.
>
