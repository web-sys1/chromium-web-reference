Fix memory leak in HatsNotificationControllerTest.

There's a circular dependency between the HatsNotificationController and
its Notification, where the controller holds a unique_ptr to the
Notification, and the notification holds a scoped_refptr to the
controller (as a notification delegate).

Code where controller creates and owns a Notification:
https://source.chromium.org/chromium/chromium/src/+/main:chrome/browser/ash/hats/hats_notification_controller.cc;l=244;drc=e5a38eddbdf45d7563a00d019debd11b803af1bb

Code where Notification holds a scoped_refptr to the controller:
https://source.chromium.org/chromium/chromium/src/+/main:chrome/browser/ash/hats/hats_notification_controller.cc;l=252;drc=e5a38eddbdf45d7563a00d019debd11b803af1bb

This circular dependency is only broken when the notification is removed
and destroyed, which happens when a user dismisses the notification.
However, in the unittest, the notification was dismissed incorrectly, so
the circular dependency remains, leaving the controller orphaned.

The fix is to dismiss the notification properly.

Repro:
1) Set up lsan build: https://www.chromium.org/developers/testing/leaksanitizer/
2) Build and run `unit_tests` and filter for HatsNotificationControllerTest:

$ autoninja -C out/lsan unit_tests && ASAN_OPTIONS="detect_leaks=1 symbolize=1 external_symbolizer_path=$PWD/third_party/llvm-build/Release+Asserts/bin/llvm-symbolizer detect_odr_violation=0" out/lsan/unit_tests --gtest_filter="HatsNotificationControllerTest.*"


This change also includes some correctness fixes:
* When move-contructing a StubNotificationDisplayService, also move the
notification.
* Fix StubNotificationDisplayService::FindNotification to not make an
unnecessary string copy.
