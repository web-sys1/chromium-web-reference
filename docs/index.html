Fenced frames: Third refactor for urn:uuid properties flow

Adds controls for visibility of fenced frame config fields to different
entities (kinds of renderers) and unifies browser->renderer information
flow for them.

After this CL:

There are two entities:
* kEmbedder (the page embedding the fenced frame)
* kContent (the page inside the fenced frame)

There are two visibility levels:
* kOpaque (must be concealed from the web platform)
* kTransparent (may be exposed to the web platform)

The API that generates a fenced frame config declares the visibility of
its fields to different entities as necessary for its privacy
guarantees. When the browser communicates information about the fenced
frame config to different entities (renderer processes), it now can
systematically redact the information according to these declarations.

The information flow goes:

1. content::browser::FencedFrameConfig
/ content::browser::FencedFrameProperties (C++)

2. blink::FencedFrame::RedactedFencedFrameConfig
/ blink::FencedFrame::RedactedFencedFrameProperties (C++),
  where content::browser::FencedFrame(Config|Properties)
  ::RedactFor(entity) performs the redaction

3. blink::mojom::FencedFrameConfig
/ blink::mojom::FencedFrameProperties, through a type mapping
