Add reporting client null check and event check to the reporting router

Previously the event reporting methods are invoked with the assumption
that the `reporting_client` has been properly initialized. While it is
usually true in the real world, the `reporting_client` could be a null
pointer, if the `RealtimeReportingClientFactory` is not bound to a
testing factory (i.e. when reporting is not the test focus).

Because `reporting_client`, which, like `safe_browsing_event_router`,
has the `ServiceIsNULLWhileTesting()` set to true, the null pointer
issue only surfaces, when we remove the now obsolete safe browsing
router null check from line 175 in the deep_scanning_utils.

This change is to make sure the reporting router, the wrapper class
around `reporting_client`, handles the null pointer cases properly.
