Disable DLL preread when there is no seek penalty or on startup

Motivation:

- From a Beta experiment, disabling DLL prereading seems to improve
  page load time. However, while there are no regressions for clients
  that have no seek penalty for chrome.dll, there are congestion and
  input delay regressions for clients with a seek penalty for
  chrome.dll. That's why we want to experiment with disabling DLL
  pre-reading for clients with no seek penalty only
  ("kNoPreReadMainDllIfSsd").

  In this experiment, we will also not include users with chrome on a
  removable drive, though this is on the roadmap for the future. More
  metrics are currently being collected to allow informed decisions
  on this front.

  For context, see crbug.com/325307453

- From a Stable experiment, disabling DLL prereading in child
  processes but enabling it in the browser process (on process launch,
  from Thread Pool, max once per 5 seconds, see
  "PrefetchVirtualMemoryPolicy") improves startup time (on clients
  with or without a seek penalty for chrome.dll) but causes many
  regressions. Since the browser process already pre-reads chrome.dll,
  it makes sense that having less pre-reading during startup
  (max 1 per 5 seconds instead of 1 for every child process launch)
  improves startup time (the DLL is probably already in physical
  memory during startup so pre-reading is just unnecessary overhead).
  That's why we want to experiment with disabling DLL pre-reading
  during startups, for all clients ("NoPreReadMainDllStartup").

Based on analysis from olivierli, spvw, etienneb and fdoray.

Verified to be working locally.
