[Tab Group Sync] Restore file:// URLs correctly on session restoration.

A recent security hardening CL (crrev.com/c/7738373) introduced a
regression where valid local URLs (like file:// URLs for local PDFs) in
saved tab groups were being overwritten with the New Tab Page on startup
(during session restoration).

This happened because the CL enforced a strict HTTP-only URL validation
during proto-to-model conversion (DataToSavedTabGroupTab), which is used
when loading from the local database.

To fix this, we now sanitize incoming remote sync data at the boundary
in AddDataToLocalStorage before it is written to the local database.
This ensures that the local database remains clean of unsafe remote
data, allowing us to safely trust the database when loading.

Consequently, we introduce IsURLValidForLocalTab, which permits
HTTP/HTTPS, NTP, and file:// URLs. We use this helper in
DataToSavedTabGroupTab to preserve valid local URLs like file:// when
loading from the DB, while still falling back to NTP for invalid or
unsafe internal URLs (like chrome://history). Similarly, the UI-level
checks in LocalTabGroupListener and SavedTabGroupWebContentsListener are
updated to use this helper to allow navigating to and opening local
file:// URLs.

Corresponding unit tests have been added to verify that file:// URLs are
preserved during local conversions and that remote file:// URLs are
correctly sanitized during sync ingestion.
