Fix PasswordAutofillAgent::ShowSuggestionsForDomain

The contract for PasswordAutofillAgent::ShowSuggestionsForDomain is to
"return true if any suggestions were shown, false otherwise". This
contract was violated in case
```
   !element.IsTextField() || !IsElementEditable(element) ||
   (!password_element.IsNull() && !IsElementEditable(password_element))
```
evaluated to true. In this case the function returned `true` without
showing password suggestions.

As a result of that, also the autofill fallback for email addresses for
the field was blocked.

This broke autofilling an email on the https://www.lufthansa.com/ ->
"Login & Registration" page.

After this CL the password manager is still unable to suggest a username
but the autofill fallback works again. It is up to the password manager
team to decide whether to relax the constraint that the non-focused
field (password or username) needs to be editable.

Low-Coverage-Reason: HARD_TO_TEST This should be tested but we don't have a test framework for PasswordAutofillAgent in place that would make it easy to test at the moment.
