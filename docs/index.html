viz: Store incremental frame damage

Previous implementation stores damage per buffer, and assumes GPU thread
uses buffers in a circular order.

Change to store incremental damage between frames instead. Main
difference is union operation happens when computing the damage for next
frame, instead of storing damage for a new frame. There is a corner case
that there is full frame damage before there are enough frames swapped
after reallocating buffers.

The result should be identical to the previous implementation. The
benefit is the new implementation does not assume a fixed number of
buffers, and instead can allow the GPU thread to dynamically allocate
buffers when needed. If there are fewer buffers, viz will just union
damage across more frames than necessary, which is still correct, though
not as efficient.
