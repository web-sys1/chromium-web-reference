`CHECK` no new ref-counts after deciding to destroy RenderProcessHost.

After a certain point in `RenderProcessHostImpl::Cleanup` a no-`return`
decision is made: a decision to delete the `RenderProcessHostImpl`
object (by posting a `DeleteSoon` task later in the `Cleanup` method).
Once that decision is made, the fate of the `RenderProcessHostImpl`
object is sealed - attempt to add new ref-counts is futile.

This CL adds a `CHECK(!deleting_soon_)` to a few methods that can
add a ref-count to `RenderProcessHostImpl`:

* `RenderProcessHostImpl::IncrementKeepAliveRefCount`
* `RenderProcessHostImpl::IncrementWorkerRefCount`
* `RenderProcessHostImpl::AddRoute`

Note that `RenderProcessHostImpl::DelayProcessShutdown` already
`return`s early if `deleting_soon_`.

This CL is somewhat motivated by https://crbug.com/1382971, but it is a
bit of a shot in the dark.  Having the extra `CHECK`s should be useful
in general (to early catch RPHI lifetime management issues), but this CL
does *not* indicate that https://crbug.com/1382971 is somehow causesd by
RPHI UaF and/or ref-count mismanagement (it *may* be caused by these
things, but we don't really know one way or another).
