Use non-allocating PRNG in poisson_allocation_sampler

Currently switching base::RandBytes() to BoringSSL causes stack overflow
because an allocation needs a random number for PoissonAllocationSampler
while RAND_bytes() from BoringSSL can allocate. Here is the repeating
fragment of the stack trace on Linux:

```
malloc()
    at allocator_shim_override_libc_symbols.h:37
OPENSSL_malloc()
    at third_party/boringssl/src/crypto/mem.c:156
RAND_bytes_with_additional_data()
    at third_party/boringssl/src/crypto/fipsmodule/rand/rand.c:371
RAND_bytes()
    at third_party/boringssl/src/crypto/fipsmodule/rand/rand.c:469
base::RandUint64()
    at base/rand_util.cc:21
base::RandDouble()
    at base/rand_util.cc:39
base::PoissonAllocationSampler::GetNextSampleInterval()
    at poisson_allocation_sampler.cc:257
base::PoissonAllocationSampler::DoRecordAlloc()
    at poisson_allocation_sampler.cc:323
base::allocator::dispatcher[...]::AllocFn()
    at base/allocator/dispatcher/dispatcher.cc:31
ShimMalloc()
    at allocator_shim.cc:200
malloc()
    at allocator_shim_override_libc_symbols.h:37
```

Expose a minimal interface necessary for this specific use:
base::RandDoubleAvoidAllocation(). Special care is taken to use the
more performant primitives for it, to avoid regressions when the sampler
is used.

Possible alternatives:

1. InsecureRandomGenerator

   - not thread safe, does not fit with thread safety guarantees for the
     allocator shim, would have needed patching to avoid calling the
     base::RandBytes() during initialization

2. partition_alloc::internal::base::RandDouble(), as demonstrated in
   http://crrev.com/c/3891028.

   - it is reserved for use only inside PartitionAlloc. There is a plan
     to move PartitionAlloc to a separate dynamic library (for cleaner
     injection via DYLD_INSERT_LIBRARIES), and having external
     dependencies on non-allocating RandDouble() would make this plan
     more difficult.
